# App Dockerization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Containerize `@advocate/app` so the full stack (Postgres + Redis + API) can be brought up with a single `docker compose up -d` command. Includes a multi-stage Dockerfile, compose service wiring, automatic migration application on first boot, and a health-check-gated dependency order.

**Architecture:** Multi-stage Dockerfile builds the monorepo with pnpm, compiles both packages with TypeScript, prunes dev dependencies for the runtime stage. The `api` service depends on `postgres` (with `service_healthy` condition) so Drizzle migrations never race ahead of the DB being ready. Migrations run as a one-shot init command inside the API container on startup.

**Tech Stack:** Docker multi-stage · node:22-alpine · pnpm (via `corepack`) · Docker Compose healthchecks · BullMQ-ready Redis connection

## Task 1: Root Dockerfile for the API

**Files:**
- Create: `packages/app/Dockerfile`

- [ ] **Step 1.1: Create the Dockerfile**

Create `E:/Projects/Stukans/advocate/packages/app/Dockerfile`:

```dockerfile
# ============================================
# Stage 1 — deps: install workspace dependencies
# ============================================
FROM node:22-alpine AS deps
WORKDIR /repo

# Enable pnpm via corepack (bundled with Node 22)
RUN corepack enable && corepack prepare pnpm@10.0.0 --activate

# Copy lockfile + all workspace manifests first so Docker caches the install layer
COPY pnpm-lock.yaml pnpm-workspace.yaml package.json turbo.json tsconfig.base.json ./
COPY packages/engine/package.json ./packages/engine/
COPY packages/app/package.json ./packages/app/

RUN --mount=type=cache,id=pnpm,target=/root/.local/share/pnpm/store \
    pnpm install --frozen-lockfile

# ============================================
# Stage 2 — builder: compile TypeScript
# ============================================
FROM node:22-alpine AS builder
WORKDIR /repo

RUN corepack enable && corepack prepare pnpm@10.0.0 --activate

# Bring in the full dependency tree from deps stage
COPY --from=deps /repo/node_modules ./node_modules
COPY --from=deps /repo/packages/engine/node_modules ./packages/engine/node_modules
COPY --from=deps /repo/packages/app/node_modules ./packages/app/node_modules

# Copy source
COPY . .

# Build engine first (app depends on its dist), then app
RUN pnpm --filter @advocate/engine build
RUN pnpm --filter @advocate/app build

# ============================================
# Stage 3 — runtime: slim image with only what's needed
# ============================================
FROM node:22-alpine AS runtime
WORKDIR /app

RUN corepack enable && corepack prepare pnpm@10.0.0 --activate

# Create a non-root user for the runtime
RUN addgroup -S advocate -g 1001 && \
    adduser -S advocate -u 1001 -G advocate

# Copy the monorepo manifest + workspace configs (pnpm needs them at runtime too)
COPY --from=builder /repo/package.json /repo/pnpm-lock.yaml /repo/pnpm-workspace.yaml ./

# Copy engine (only dist + manifest)
COPY --from=builder /repo/packages/engine/package.json ./packages/engine/
COPY --from=builder /repo/packages/engine/dist ./packages/engine/dist

# Copy app (dist + manifest + migrations + drizzle config)
COPY --from=builder /repo/packages/app/package.json ./packages/app/
COPY --from=builder /repo/packages/app/dist ./packages/app/dist
COPY --from=builder /repo/packages/app/drizzle ./packages/app/drizzle

# Install production deps only (re-installing here drops devDependencies)
RUN --mount=type=cache,id=pnpm,target=/root/.local/share/pnpm/store \
    pnpm install --prod --frozen-lockfile

# Ownership + non-root
RUN chown -R advocate:advocate /app
USER advocate

EXPOSE 3000

# Entrypoint: apply pending migrations, then start server.
# Using sh -c keeps it a single container responsibility — no migrate sidecar needed
# for single-instance dev/staging. Switch to a separate one-shot `migrate` service
# in production k8s so replicas don't race.
CMD ["sh", "-c", "node packages/app/dist/db/migrate.js && node packages/app/dist/server/server.js"]

HEALTHCHECK --interval=10s --timeout=3s --start-period=30s --retries=5 \
  CMD wget -qO- http://localhost:3000/health || exit 1
```

## Task 2: Compile migrate.ts Without dotenv at Container Runtime

**Files:**
- Modify: `packages/app/src/db/migrate.ts`

The current `migrate.ts` starts with `import 'dotenv/config';` — needed for local `pnpm db:apply` but harmful inside a container where env comes from docker-compose and no `.env` file exists. The import doesn't fail at runtime (dotenv tolerates a missing `.env`), but we can gate it so the container path is cleaner and doesn't log a confusing "no .env file" line.

- [ ] **Step 2.1: Update `packages/app/src/db/migrate.ts`**

Replace with:

```typescript
import { resolve } from 'node:path';
import { fileURLToPath } from 'node:url';
import { migrate } from 'drizzle-orm/node-postgres/migrator';
import { childLogger } from '../config/logger.js';
import { closeDb, getDb } from './connection.js';

// Load .env only when running outside a container. Inside Docker, env comes from
// the compose file / k8s manifest; no .env file is present and dotenv would log
// a (harmless) warning that pollutes migration logs.
if (process.env.NODE_ENV !== 'production' && !process.env.RUNNING_IN_CONTAINER) {
  await import('dotenv/config');
}

const log = childLogger('migrate');

async function run(): Promise<void> {
  const migrationsFolder = resolve(fileURLToPath(import.meta.url), '../../../drizzle/migrations');
  log.info({ migrationsFolder }, 'running migrations');
  const db = getDb();
  await migrate(db, { migrationsFolder });
  log.info('migrations complete');
  await closeDb();
}

run().catch((err) => {
  log.error({ err }, 'migration failed');
  process.exit(1);
});
```

## Task 3: Compose Service for the API

**Files:**
- Modify: `docker-compose.yml`

- [ ] **Step 3.1: Add the `api` service to docker-compose.yml**

Edit `E:/Projects/Stukans/advocate/docker-compose.yml`. Add a new `api` service block BEFORE the `volumes:` section. The full file should become:

```yaml
services:
  postgres:
    image: postgres:17-alpine
    container_name: advocate-postgres
    ports:
      - "${POSTGRES_PORT:-36432}:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: advocate
      POSTGRES_USER: advocate
      POSTGRES_PASSWORD: advocate
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "advocate"]
      interval: 10s
      timeout: 3s
      retries: 5
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  redis:
    image: redis:7-alpine
    container_name: advocate-redis
    ports:
      - "${REDIS_PORT:-36479}:6379"
    volumes:
      - redisdata:/data
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 3s
      retries: 5
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  api:
    build:
      context: .
      dockerfile: packages/app/Dockerfile
      target: runtime
    container_name: advocate-api
    ports:
      - "${API_PORT:-36401}:3000"
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
    environment:
      NODE_ENV: production
      RUNNING_IN_CONTAINER: "true"
      API_PORT: "3000"
      # Container-internal hostnames (service names); host port vars are only
      # used by the host when connecting from outside the compose network.
      DATABASE_URL: postgresql://advocate:advocate@postgres:5432/advocate
      REDIS_URL: redis://redis:6379
      ANTHROPIC_API_KEY: "${ANTHROPIC_API_KEY:-}"
      GOOGLE_AI_API_KEY: "${GOOGLE_AI_API_KEY:-}"
      OPENAI_API_KEY: "${OPENAI_API_KEY:-}"
      DEEPSEEK_API_KEY: "${DEEPSEEK_API_KEY:-}"
      QWEN_API_KEY: "${QWEN_API_KEY:-}"
      LLM_MONTHLY_BUDGET_CENTS: "${LLM_MONTHLY_BUDGET_CENTS:-2000}"
      LLM_DEFAULT_MODE: "${LLM_DEFAULT_MODE:-balanced}"
      KEYCLOAK_URL: "${KEYCLOAK_URL:-http://host.docker.internal:9080}"
      KEYCLOAK_REALM: "${KEYCLOAK_REALM:-advocate}"
      KEYCLOAK_CLIENT_ID: "${KEYCLOAK_CLIENT_ID:-advocate-app}"
      TELEGRAM_BOT_TOKEN: "${TELEGRAM_BOT_TOKEN:-}"
      TELEGRAM_CHANNEL_ID: "${TELEGRAM_CHANNEL_ID:-}"
      CREDENTIAL_MASTER_KEY: "${CREDENTIAL_MASTER_KEY}"
      LOG_LEVEL: "${LOG_LEVEL:-info}"
    healthcheck:
      test: ["CMD", "wget", "-qO-", "http://localhost:3000/health"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 30s
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

volumes:
  pgdata:
  redisdata:
```

Key points:
- `api` depends on `postgres:service_healthy` and `redis:service_healthy` — the API container won't start until both DBs are healthy
- Env vars use compose network hostnames (`postgres`, `redis`), NOT `localhost` — localhost inside the API container is the API container itself, not the DB
- `CREDENTIAL_MASTER_KEY` has no `:-` default because a missing master key should fail fast (env validation will reject at boot)
- `host.docker.internal` is used for Keycloak since it lives on the host machine, not in the compose network
- Non-LLM keys have `:-` defaults to empty string (validated as `.min(1).optional()` → undefined inside the app)
