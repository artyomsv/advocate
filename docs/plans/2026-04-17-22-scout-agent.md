# Plan 22 ‚Äî Scout agent (thread discovery)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development.

**Goal:** Scout periodically scans configured subreddits, LLM-scores threads for product relevance, and seeds the orchestrator with promising ones. Tag `plan22-complete`.

**Prerequisites:** Plan 20 (Reddit OAuth), Plan 21 (RedditClient).

## Architecture

- `RedditClient.fetchListing(legendAccountId, subreddit, sort, limit)` ‚Üí list of thread records (id, title, body, upvotes, created, url).
- `Scout.scan(options)` ‚Äî for a single product + community:
  1. Picks the first active legend for the community's platform; gets its account.
  2. Fetches `r/<community.identifier>/hot?limit=25`.
  3. For each thread, LLM-scores relevance 0-10 against product's value props + audiences. Cheap Gemini Flash call, batchable prompt.
  4. Threads scoring ‚â• threshold (default 7): enqueue `orchestrate` job with `threadContext` set to the thread's title + snippet.
- Queue: `scout.scan`; job data `{ productId, communityId }`. Worker consumes + calls `Scout.scan`.
- Cron: dashboard schedules or a per-product default `*/30 * * * *` via BullMQHeartbeatScheduler.

## Tasks

### T1 ‚Äî fetchListing on RedditClient

Add method. Returns normalized `RedditThread[]` with `{ id, title, body, url, score, createdUtc, isSelfPost, numComments }`. Reddit's listing response shape: `{ data: { children: [{ kind, data: {...} }] } }`.

Parse each child to a RedditThread; filter out stickied + NSFW + promoted.

Unit tests mock fetch + assert parser handles edge cases.

### T2 ‚Äî Scout class + scoring prompt

`Scout.scanAndDispatch(deps: { productId, communityId })`:

1. Load product + community from db (communityId tells us subreddit name).
2. Load first `active` legend for that product + community's platform; load its legend_account.
3. `fetchListing('hot', 25)`.
4. LLM-score via `router.invoke({ taskType: 'classification', ... })` with a batched prompt listing all 25 threads; expect JSON `{ scores: { "<threadId>": 0-10 } }`.
5. Threads ‚â• threshold (default 7, configurable via scout_config): enqueue `orchestrate` job with `threadContext = title + "\n\n" + body.slice(0,500)`.
6. Log count discovered + count dispatched.

No new DB writes in MVP ‚Äî the orchestrator downstream creates content_plans. Future: a `discoveries` table with score + disposition history.

### T3 ‚Äî scout.scan worker + BullMQ queue + schedule helper

- Add `scout.scan` to `QUEUE_NAMES`
- `createScoutWorker` factory
- Launch in worker.ts when Reddit config exists
- Helper: `scheduleScoutForCommunity(productId, communityId, cronPattern)` via existing `BullMQHeartbeatScheduler` (Plan 11e)

### T4 ‚Äî Verify + tag

Typecheck, unit tests (listing parser + score prompt shape), Docker smoke (worker logs scout listener), tag `plan22-complete`.

Live flow owner manual: connect legend ‚Üí schedule scout for product+community ‚Üí wait ‚Üí check /agents/activity for newly-surfaced content_plans with threadContext populated.

## Out of scope

- Thread de-duplication (seen-threads table) ‚Äî scout may re-surface same thread until it's acted on
- Cross-subreddit aggregation (one community per job for now)
- Comment engagement (we only find root threads)
