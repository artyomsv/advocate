# Plan 11.5 ‚Äî Engine store persistence (keystone)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development.

**Goal:** Replace the four in-memory engine stores (memory, tasks, messages, budget) with Drizzle-backed implementations in `@mynah/app`. The engine keeps its in-memory variants for tests; the app uses the persistent ones at runtime. Unlocks real agent traces, kanban UI, durable inter-agent memory, and cost attribution over time. Tag `plan11.5-complete`.

**Prerequisites:** Plans 01-23 + R + 25. All Drizzle tables already exist from Plan 02.

## Architecture

- **New module:** `packages/app/src/engine-stores/` holds the four Drizzle implementations.
- Each class implements the interface exported from `@mynah/engine`.
- App boot wires `AgentRuntime` + `LLMRouter` with Drizzle-backed instances instead of in-memory ones.
- Existing in-memory variants stay in `@mynah/engine` for unit tests ‚Äî no change to the engine package.
- Message bus + log: separate reads (`queryLog`) from writes (`publish`) ‚Äî both back onto `agent_messages` table.
- Budget tracker: aggregates from `llm_usage` instead of keeping a mutable counter. `recordSpend` is a no-op because `llm_usage` rows are already written by providers.

- `clear(agentId)` ‚Äî DELETE

Same pattern for relational + consolidated. Ensure `agentId` is the Drizzle-side UUID type (branded), match the engine's `AgentId`.

### T3 ‚Äî Message bus + conversation log

**Files:**
- `packages/app/src/engine-stores/messaging/drizzle-message-bus.ts`
- `packages/app/src/engine-stores/messaging/drizzle-conversation-log.ts`
- `packages/app/tests/engine-stores/messaging.test.ts`

`DrizzleMessageBus` implements `MessageBus`:
- `publish(msg)` ‚Äî INSERT into `agent_messages` ‚Üí fan out to in-process subscribers
- `subscribe(topic, handler)` ‚Äî in-process only (process-local), returns unsubscribe
- Note: cross-process pub-sub (worker fires, API renders) would require LISTEN/NOTIFY or Redis pubsub. Out of scope for this plan ‚Äî just DB write + single-process subscribers. The important invariant is that every message is **persisted**, so `/agents/activity` can read the whole history.

`DrizzleConversationLog` implements `ConversationLog`:
- `append(msg)` ‚Äî INSERT (same table as bus writes)
- `readByCorrelation(correlationId)` ‚Äî SELECT ... WHERE correlation_id = $1 ORDER BY created_at
- `readByAgent(agentId, limit?)` ‚Äî SELECT by sender_agent_id

**Important:** make `DrizzleMessageBus` and `DrizzleConversationLog` share a write path so a single `publish` produces one DB row visible to both.

**Commit:**
```
feat(engine-stores): Drizzle-backed message bus + conversation log
```

### T4 ‚Äî Budget tracker (aggregates from llm_usage)

**Files:**
- `packages/app/src/engine-stores/budget/drizzle-budget-tracker.ts`
- `packages/app/tests/engine-stores/budget.test.ts`

`DrizzleBudgetTracker` implements `BudgetTracker`:
- `recordSpend(providerId, model, millicents)` ‚Äî **no-op**. Provider code already INSERTs a `llm_usage` row with the cost; recording twice would double-count.
- `getSpend(window) ‚Üí { totalMillicents, perProvider }` ‚Äî SELECT SUM(cost_millicents) FROM llm_usage WHERE created_at >= window-start
- `isOverBudget(capCents) ‚Üí boolean` ‚Äî `totalMillicents / 1000 >= capCents`

The in-memory variant (from `@mynah/engine`) tracks a running counter. The Drizzle variant queries on demand. Callers don't care.

**Commit:**
```
feat(engine-stores): Drizzle-backed budget tracker (aggregates llm_usage)
```

## Out of scope

- Cross-process pub-sub (LISTEN/NOTIFY / Redis pubsub for MessageBus)
- Memory consolidation job (there's a `MemoryConsolidator` interface ‚Äî actually running it on a cron is a separate plan)
- Backfill: existing in-memory state at boot is empty in either variant, so no migration needed
- Secondary indexes beyond what the schema already has (`agentId`, `createdAt`, `correlationId`)
